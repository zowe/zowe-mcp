# Zowe MCP - Agent Instructions

## Project Overview

Zowe MCP is a Model Context Protocol (MCP) server and VS Code extension that provides AI tools for interacting with z/OS systems. It enables LLMs to work with data sets, jobs, and UNIX System Services on one or more z/OS systems.

## Repository Structure

This is an npm workspaces monorepo with four packages:

- `packages/zowe-mcp-common` — Shared utilities (CJS, consumed by all other packages). Exports `plural()`.
- `packages/zowe-mcp-server` — Standalone MCP server (ESM, npm **`@zowe/mcp-server`**)
- `packages/zowe-mcp-vscode` — VS Code extension that registers the server (CommonJS) and contributes Zowe and ISPF color themes (`themes/*.json`)
- `packages/zowe-mcp-evals` — AI evaluations: runs an LLM agent against the MCP server (mock or native), checks tool choice/arguments and answer content, produces a Markdown report. Uses Vercel AI SDK and MCP SDK client. Config: gitignored `evals.config.json` at **repo root** (vLLM, Gemini, or LM Studio). Supports single-model (legacy) or **multi-model** (`models` array with `id` per entry; first is default). Use `--model <id>` to select a model when using multi-model config. LM Studio provider (`"lmstudio"`) uses the OpenAI-compatible endpoint (default `http://localhost:1234/v1`); the runner auto-loads the model via `POST /api/v1/models/load` with `contextLength` (default 32768) so the context window is large enough for tool definitions; when `serverModel` is missing, `loadEvalsConfig` queries `GET /v1/models` and lists available models in the error message. Run from repo root: `npm run evals` (options after `--`). Question sets: YAML files in `questions/` with per-set `repetitions`, `minSuccessRate`, optional `mock`/`native`, optional `systemPrompt`/`systemPromptAddition`, optional `skip` (string reason). Individual questions also support `skip` (string reason). Skipped sets/questions are logged but not executed or counted. Assertions use Ansible-style key-based format (3 types): **toolCall** (unified: tool/tools/oneOf, optional count/minCount/args), **toolCallOrder** (ordered sequence, value is direct array of steps), **answerContains** (substring or pattern). Each assertion can have an optional `name` for failure messages. Composites `allOf`/`anyOf` for logical grouping. **validDsn** is a special key inside `toolCall.args` (or `toolCallOrder` step `args`) whose value is a canonical DSN string (e.g. `USER.SRC.COBOL(CUSTFILE)`); the runner accepts any server-equivalent form (quoted, parenthesized, separate dsn+member). A tool DSN registry (`src/tool-dsn-registry.ts`) maps each tool to its dsnParam/memberParam; using `validDsn` for an unregistered tool fails fast. DSN normalization lives in `src/dsn-utils.ts`.

## Key Architectural Decisions

- **npm package name (MCP server)**: Published as **`@zowe/mcp-server`**. The CLI executable name remains **`zowe-mcp-server`** (see `bin` in the server `package.json`), so `npx @zowe/mcp-server` and the global binary `zowe-mcp-server` behave the same after install.
- **CI npm tarball**: GitHub Actions runs `npm pack -w @zowe/mcp-server` after tests pass and uploads the tarball as artifact **`zowe-mcp-server-npm`** (filename pattern `zowe-mcp-server-<version>.tgz`). Locally: `npm run pack:server` writes the same file to the repo root (ignored by `.gitignore`). The **`prepack`** script (`scripts/bundle-for-pack.cjs`) rewrites workspace deps (`zowe-mcp-common` → `.local/`) and file-based tgz deps (`zowe-native-proto-sdk` → `.unpack/`), copies the rewritten `package.json` to an isolated temp directory outside the monorepo, runs `npm install --omit=dev` there (so deps are not hoisted to the workspace root), copies the resulting `node_modules` back, and adds `bundledDependencies: true` so `npm pack` includes the full tree. The **`postpack`** script restores the original `package.json` and cleans up. **Important**: `bundledDependencies` must NOT be in the committed `package.json` — it would cause `npm install` to skip deps during development (npm marks them as `inBundle` in the lockfile). The prepack script adds it dynamically. Both scripts use shared helpers from `scripts/bundle-production-deps.cjs` (also used by the VSIX `bundle-server.js`). Test airgapped installation with `npm run test:airgap` (uses empty cache, invalid registry, 5ms timeout), `npm run test:airgap:build` (builds and packs first), or `npm run test:airgap:build:native` (also runs a native z/OS smoke test using `native-config.json` and `.env` from the repo root).
- **Standalone MCP clients (e.g. Roo Code)**: Some clients only read their own MCP config (e.g. `.roo/mcp.json`) and do not use VS Code’s registered MCP providers. Use **`@zowe/mcp-server`** in stdio mode with `--native` / `--mock`, env passwords (`ZOWE_MCP_PASSWORD_*`), optional `--config` for job cards. See **`docs/roo-or-standalone-mcp.md`** and **`docs/examples/roo-mcp.json`**.
- **Monorepo with npm workspaces**: Four packages share a root `package.json`. The `zowe-mcp-common` package (CJS) provides shared utilities (`plural()`) consumed by server, extension, and evals. The VS Code extension depends on the server and common packages via workspace linking. npm workspaces respects the dependency topology — common builds first.
- **npm registry (Zowe Artifactory)**: The project uses Zowe Artifactory for npm installs (`.npmrc`: `registry=https://zowe.jfrog.io/artifactory/api/npm/npm-release/`). This supplies `@zowe/*` packages. The `zowe-native-proto-sdk` is always a `file:` dependency pointing to a versioned tarball in `resources/` (e.g. `file:../../resources/zowe-native-proto-sdk-0.3.0.tgz`). All `sdk:*` scripts download/copy the SDK to `resources/zowe-native-proto-sdk-<version>.tgz` and update the server's `package.json` dependency accordingly. Use `npm run sdk:fallback`, `sdk:release`, `sdk:nightly`, `sdk:pr`, `sdk:branch`, or `sdk:local` to switch between sources (see SDK switching). The `deps/` directory is legacy (gitignored). The committed SDK in `resources/` is the baseline for CI so the build does not depend on zowe-native-proto's z/OS build on Marist.
- **Transport separation**: Server logic (`server.ts`) is transport-agnostic. Transport implementations live in `src/transports/` (stdio and HTTP Streamable). The entry point (`index.ts`) selects the transport based on CLI args.
- **CLI parsing (yargs)**: The server uses **yargs** (same as Zowe Imperative) for CLI argument parsing and help. Options and subcommands are defined in `index.ts`; `-h` / `--help` and `--version` are generated by yargs. Subcommands `init-mock` and `call-tool` delegate to scripts via `spawnSync`; env overrides (e.g. `ZOWE_MCP_MOCK_DIR`) are applied after parsing.
- **HTTP multi-session**: The HTTP transport creates a new `McpServer` + `StreamableHTTPServerTransport` per client session. `startHttp` accepts a server factory (`() => McpServer`) rather than a single server instance. Sessions are tracked by `mcp-session-id` header and cleaned up on close. Default HTTP port is **7542** (Zowe 75xx range; Zowe API ML uses 7552–7558).
- **Component-based tools**: Tools are organized under `src/tools/<component>/`. Each component registers its tools via a function that takes an `McpServer` instance. Components: `context` (system management and server info — `getContext` is always registered, `listSystems`/`setSystem` only when a backend is present), `datasets` (data set CRUD, restoreDataset), `uss` (USS list/read/write/create/delete/chmod/chown/chtag/copyUssFile, runSafeUssCommand, temp tools, getUssHome, changeUssDirectory), `local-files` (downloadDatasetToFile, uploadFileToDataset, downloadUssFileToFile, uploadFileToUssFile, downloadJobFileToFile — workspace paths validated via MCP `roots/list` when the client supports it, else `ZOWE_MCP_WORKSPACE_DIR`, `ZOWE_MCP_LOCAL_FILES_ROOT`, or `--local-files-root`), `tso` (runSafeTsoCommand), `console` (runConsoleCommand — **disabled**: ZNP does not yet support `console.issueCmd`; code is kept, registration commented out in `server.ts`), `jobs` (submitJob, getJobStatus), `zowe-explorer` (openDatasetInEditor, openUssFileInEditor, openJobInEditor — registered when Zowe Explorer is available and extension pipe is connected). Use `server.registerTool(name, config, cb)` — the older `server.tool(...)` overloads are deprecated.
- **Tool naming for Copilot**: MCP tool names use **camelCase** (e.g. `listDatasets`, `setSystem`, `getContext`) so they fit VS Code Copilot’s display. Copilot prefixes names with `mcp_<providerId>_`, so a tool named `listDatasets` appears as `mcp_zowe_listDatasets`. Keep names short and avoid redundant `zowe_` prefixes — the provider ID already provides the namespace.
- **ESM for server, CJS for extension and common**: The MCP SDK requires ESM. VS Code extensions use CommonJS. The `zowe-mcp-common` package outputs CJS so both ESM (server, evals) and CJS (extension) consumers can import it. Each package has its own `tsconfig.json`.
- **Version from package.json**: The server reads its version from `package.json` at runtime using `createRequire`. Keep the version in `package.json` as the single source of truth.
- **Code duplication detection (jscpd)**: The `jscpd` tool (Rabin-Karp token-based copy/paste detector) is configured via `.jscpd.json` at the repo root. Run `npm run duplication` to scan all packages. A Cursor hook (`.cursor/hooks/duplication.sh`) runs jscpd on edited `.ts` files after Agent edits to warn about introduced duplication. Config: `minLines: 10`, `minTokens: 50`, `threshold: 5%`; excludes tests, themes, build artifacts, and `.d.ts` files.
- **Code formatting via Cursor hooks**: All TypeScript, JavaScript, and JSON files are formatted with Prettier (`.prettierrc.json` + `prettier-plugin-organize-imports`). Markdown files are formatted with markdownlint-cli2. A Cursor hook (`.cursor/hooks/format.sh`) auto-formats all file types after every Agent and Tab edit.
- **ESLint with type-checked rules**: ESLint is configured with `typescript-eslint`'s `recommendedTypeChecked` + `stylisticTypeChecked` rulesets, `eslint-plugin-headers` for license headers, and `eslint-plugin-vitest` for test hygiene (server tests only — VS Code extension tests use Mocha). Config is in `eslint.config.mjs`. Each package has a `tsconfig.eslint.json` that includes all lintable files (src, tests, scripts). The Cursor format hook automatically runs `eslint --fix` on `.ts` files.
- **License header enforcement**: All `.ts` files must start with the EPL-2.0 license header. Enforced by `eslint-plugin-headers` via `eslint.config.mjs`. The Cursor format hook automatically inserts missing headers on save. Run `npm run lint` to check all files, `npm run lint:fix` to auto-fix.
- **Structured logging (MCP server)**: The server uses a custom `Logger` class (`src/log.ts`) that writes human-readable messages to stderr and forwards them to the MCP client via `sendLoggingMessage()`. Log levels follow RFC 5424 (debug, info, notice, warning, error, critical, alert, emergency). The `logging` capability is declared in `server.ts` so the SDK allows protocol-level log notifications. No external logging library (pino, winston) is used — the MCP SDK provides the protocol transport, and stderr handles local diagnostics.
- **Event-based extension communication**: The MCP server and VS Code extension communicate bidirectionally over a named pipe (Unix socket / Windows named pipe). The extension creates a per-workspace pipe server on activation and writes a discovery file to `context.globalStorageUri`. The MCP server reads the discovery file (via `MCP_DISCOVERY_DIR` + `WORKSPACE_ID` env vars) and connects. Events are framed as newline-delimited JSON (NDJSON). Event types are defined in `src/events.ts` and split into `ServerToExtensionEvent` (e.g. `log`, `notification`, `request-password`, `password-invalid`, `store-password`) and `ExtensionToServerEvent` (e.g. `log-level`, `password`, `connections-update`, `zowe-explorer-update`). When the env vars are absent (standalone mode), the pipe client is not created and the server operates identically to before.
- **Password collection (native backend)**: When the VS Code extension is connected, the server uses the **pipe only** (`request-password` → extension prompts with masked input or SecretStorage → `password` event). When the extension is not connected, MCP elicitation is used when the client supports it. The elicitation schema hints that clients should use a masked input (some clients still show plain text). When multiple tools request credentials for the same connection concurrently, only one prompt runs and the others wait. While waiting for a password, the credential provider reports progress ("Waiting for password") when the caller passes `options.progress` to `getCredentials`. After successful use of an elicited password, the server sends `store-password` so the extension can persist it. The extension command "Zowe MCP: Clear Stored Password" clears the stored password. Note: the MCP spec requires URL mode for sensitive data; this implementation uses form mode for compatibility with form-only clients.
- **Backend selector (VS Code setting)**: The `zoweMCP.backend` dropdown setting (`"native"` | `"mock"`, default `"native"`) controls which backend the MCP server uses. When `"native"`, the extension passes `--native` with connection specs from `zoweMCP.nativeConnections`. When `"mock"`, the extension passes `--mock <dir>` using the path from `zoweMCP.mockDataDirectory`. **Auto-migration**: if `backend` has never been set by the user and `mockDataDirectory` is non-empty and `nativeConnections` is empty, the extension auto-migrates `backend` to `"mock"` (Global scope). **Prompt-on-select**: when the user switches `backend` to `"mock"` and no `mockDataDirectory` is configured, the extension offers to generate or select mock data. Changes require restarting the MCP server (reload window).
- **Generate Mock Data command**: The `zowe-mcp.initMockData` VS Code command (palette: "Zowe MCP: Generate Mock Data") runs the bundled server's `init-mock` script. It prompts for an output folder, generates mock data, offers to set `zoweMCP.mockDataDirectory`, and offers to reload the window. This is the primary way for extension users to create mock data (they cannot use `npx @zowe/mcp-server` since the server is bundled inside the extension).
- **Dynamic log level from VS Code**: The `zoweMCP.logLevel` VS Code setting controls the MCP server's log verbosity at runtime. Changes are sent as `log-level` events over the named pipe and take effect immediately without restarting the server. The initial level is sent when the server first connects.
- **Tool-call logging (middleware)**: When enabled, every MCP tool call is logged with full input and full response (and backend type). Controlled by `CreateServerOptions.logToolCalls` or env `ZOWE_MCP_LOG_TOOL_CALLS=1`/`true`. When enabled, `createServer()` patches `server.registerTool` so each invocation logs "Tool call" (tool name, backend, full arguments), then on success "Tool call result" (tool, backend, full result) or on error "Tool call error" (tool, backend, error message). Logs go to the `tools` child logger (stderr and MCP/pipe when connected). No abbreviation—full request and response for debugging.
- **ZosBackend interface**: All z/OS data set operations go through the `ZosBackend` interface (`src/zos/backend.ts`). This abstraction boundary decouples tools/resources from the actual z/OS API. Any backend (z/OSMF, Zowe SDK, mock filesystem) can be plugged in. The tool and resource layer is completely backend-agnostic.
- **Per-system working context**: Each z/OS system maintains its own `SystemContext` (user ID) in a `SessionState` (`src/zos/session.ts`). Switching systems restores that system's context — like separate terminal sessions per machine.
- **DSN resolution**: Data set names are always fully qualified. Input may optionally be wrapped in single quotes (e.g. `'USER.SRC.COBOL'`); quotes are stripped. The `resolveDsn(input, member?)` and `resolvePattern(input)` functions in `src/zos/dsn.ts` handle normalization, validation (44-char limit, 8-char qualifiers), and case normalization to uppercase. MY.DATASET and 'my.dataset' are equivalent.
- **Mock mode**: The server supports a filesystem-backed mock mode (`--mock <dir>` or `ZOWE_MCP_MOCK_DIR` env var). The `FilesystemMockBackend` (`src/zos/mock/`) implements `ZosBackend` using a DSFS-inspired directory layout. Mock data is generated by `init-mock` CLI (`npx @zowe/mcp-server init-mock --output ./zowe-mcp-mock-data`). ETags use file mtime.
- **Native (SSH) backend**: The server can use the Zowe Native Proto SDK (`src/zos/native/`) to connect to z/OS over SSH. Connection format is `user@host` or `user@host:port`. Only `listDatasets` is implemented; other backend methods throw "not implemented". The native backend calls `listDatasets` with `attributes: true` by default so list responses include data set attributes (dsorg, recfm, lrecl, etc.); the tool’s `detail` parameter (`minimal`, `basic`, `full`; default `basic`) controls which fields are returned to the LLM (see detail levels and VSAM handling in Things to Remember). Standalone: systems from `--config <path>` (JSON `{ "systems": ["user@host", ...] }`) or repeatable `--system`; passwords from env `ZOWE_MCP_PASSWORD_<USER>_<HOST>`. VS Code: connection list from `zoweMCP.nativeConnections` setting; passwords via pipe only when the extension is connected; otherwise via elicitation when the client supports it. Elicited passwords are persisted via store-password event. Extension command "Zowe MCP: Clear Stored Password" clears SecretStorage for a chosen connection. Shared secret key: `zowe.ssh.password.${user}.${hostNormalized}` (Zowe OSS convention). Invalid passwords are blacklisted (standalone) or deleted from storage (VS Code). ZNP response timeout (seconds, default 60): CLI `--native-response-timeout`, env `ZOWE_MCP_NATIVE_RESPONSE_TIMEOUT`, VS Code `zoweMCP.nativeResponseTimeout`; changes sent as `native-options-update` and apply to future connections.
- **Tool annotations**: Read-only tools use `readOnlyHint: true` (skips VS Code confirmation dialog). Destructive tools (`deleteDataset`) use `destructiveHint: true`. This follows VS Code MCP best practices.
- **MCP output schemas**: Tools declare a Zod `outputSchema` so `tools/list` advertises the response shape and clients can validate `structuredContent`. Each component that has tools with envelope-shaped or structured responses has a `*-output-schemas.ts` file (e.g. `dataset-output-schemas.ts`, `context-output-schemas.ts`, `uss-output-schemas.ts`, `tso-output-schemas.ts`, `jobs-output-schemas.ts`). Success responses use `wrapResponse()` which returns both `content` (JSON text) and `structuredContent` (the same envelope); the schema must match that structure. For **tool execution errors** (validation failure, blocked command, user decline), return `{ content: [...], isError: true }` so the result is not required to conform to the success schema (per MCP spec). The Zowe Explorer open-in-editor tools (`openDatasetInEditor`, `openUssFileInEditor`, `openJobInEditor`) do not have output schemas and return plain text only.
- **MCP progress**: When the client sends `_meta.progressToken` in `tools/call`, tools report progress via `createToolProgress(extra, title)` in `src/tools/progress.ts`. Each tool has a human-friendly title (e.g. "List members of {dsn}"). The reporter exposes `start()`, `step(subactionMessage)`, and `complete(finalMessage)`; messages are formatted as "title - message". The backend interface has an optional `progress?: BackendProgressCallback` on I/O methods; the native backend reports subactions (e.g. "Connecting to host via SSH", "Deploying Zowe Native server to host", "Running Zowe Native operation") so the user knows why they are waiting.
- **Prompt naming**: MCP prompts use `camelCase` names per VS Code conventions (e.g. `reviewJcl`, `explainDataset`, `compareMembers`). They appear as `/mcp.zowe.<name>` slash commands.
- **Resource URI scheme**: Data set resources use `zos-ds://{system}/{dsn}` and `zos-ds://{system}/{dsn}({member})` URI templates with optional `?volser=` for uncataloged data sets.
- **No default user on system**: `ZosSystem` does not have a `defaultUser` property. The default user for a system is determined by the `CredentialProvider` (e.g. the first credential entry in mock mode, or the logged-in user from Zowe config). The `setSystem` tool calls `credentialProvider.getCredentials(systemId)` to get the default user when first connecting.
- **Auto-activation of single system**: When only one system is configured, `createServer()` automatically activates it (calls `setActiveSystem` with the default user from the credential provider). This means the LLM can skip `setSystem` and jump straight to data set tools in single-system setups.
- **Lazy context initialization**: Dataset tools lazily initialize the system context via `ensureContext()` when the LLM passes an explicit `system` parameter without first calling `setSystem`. The `DatasetToolDeps` interface includes `credentialProvider` for this purpose.
- **Connection vs system terminology**: A **connection** is one SSH access (user@host or user@host:port); what users configure. A **system** (z/OS system) is the target host. One system can have multiple connections. The `system` parameter accepts a host (when only one connection exists for that host) or a connection spec (user@host) when multiple connections exist; if only the host is given and multiple connections exist, the tool fails with "Valid values: …" listing the connection specs.
- **System parameter (host or connection spec)**: All tools and prompts that take an optional `system` parameter use `resolveSystemForTool(registry, sessionState, system)` from `src/zos/session.ts`, which returns `{ systemId, userId? }`. It accepts a hostname (FQDN or unqualified when unambiguous) or a connection spec (user@host). When multiple connections exist for a host, passing only the host throws with valid connection values. Registry entries (from native loader) include optional `connectionSpecs` per host; `listSystems` and `getContext` expose `connections` and `activeConnection`.
- **ISPF-style pattern matching**: The mock backend's `matchPattern()` function treats a trailing lone `*` as `**` (match across any number of qualifiers), following the standard ISPF 3.4 convention. So `USER.*` matches `USER.SRC.COBOL`, `USER.JCL.CNTL`, etc. A `*` in a non-trailing position still matches within a single qualifier only.
- **Response envelope pattern**: All data set tool responses are wrapped in a `ToolResponseEnvelope` (`src/tools/response.ts`) with `_context` (resolution metadata), `_result` (summary/pagination/windowing metadata), and `data` (the actual payload). The underscore prefix signals metadata. Resolved values (`resolvedPattern`, `resolvedDsn`, `resolvedTargetDsn`) are only present in `_context` when resolution actually changed the input (e.g. normalized case, stripped quotes). They are fully qualified with no quotes.
- **MCP server instructions**: The server sends `instructions` during initialization (MCP spec `ServerOptions.instructions`). The instructions describe the pagination protocol (list pagination with offset/limit, line-windowed pagination with startLine/lineCount). Clients may add this to the system prompt. Defined as `SERVER_INSTRUCTIONS` in `server.ts`.
- **Pagination for list operations**: `listDatasets` and `listMembers` support `offset` (0-based, default 0) and `limit` (default 500, max 1000) parameters. The backend returns the full list; the tool layer slices it. `_result` includes `count`, `totalAvailable`, `offset`, and `hasMore`. When `hasMore` is true, the envelope `messages` array contains a strong directive to call the tool again with the next offset/limit. Constants: `DEFAULT_LIST_LIMIT = 500`, `MAX_LIST_LIMIT = 1000`.
- **Pagination description pattern**: Tool descriptions do NOT inline the full pagination protocol. Instead, they append a short note via `withPaginationNote(description, PAGINATION_NOTE_LIST)` or `withPaginationNote(description, PAGINATION_NOTE_LINES)` from `response.ts`. The full protocol lives in `SERVER_INSTRUCTIONS` (sent to clients at init). This keeps descriptions concise and avoids duplication.
- **Response cache**: Backend results (e.g. `listDatasets`, `listMembers`, `readDataset`, `searchInDataset`, `runTsoCommand`) are cached so repeated calls with the same parameters do not hit the backend. For `searchInDataset`, the full search result is cached (key: systemId, dsn, member, string, parms, encoding); pagination slices the cached members. For `readDataset`, the full content is cached and line windowing is applied in the tool layer, so pagination does not re-read from the backend. **Storage**: in-memory only (LRU, not persisted). Each server instance gets a new empty cache unless you pass a `ResponseCache` instance. Tests can pass `responseCache: createResponseCache()` to guarantee an empty cache. Cache key: built by caller via `buildCacheKey(prefix, params)` (params as key-value record, keys sorted for stability). Default TTL 10 minutes, default max size 1 GB; configurable via `CreateServerOptions.responseCache` (and optionally CLI: `--response-cache-ttl-minutes`, `--response-cache-max-mb`, `--response-cache-disable`, or env: `ZOWE_MCP_RESPONSE_CACHE_TTL_MINUTES`, `ZOWE_MCP_RESPONSE_CACHE_MAX_BYTES`, `ZOWE_MCP_RESPONSE_CACHE_DISABLE`). Legacy env `ZOWE_MCP_RESPONSE_CACHE_TTL_MS` (milliseconds) is still supported. Set `responseCache: false` to disable. Implementation: `src/zos/response-cache.ts` (generic `createResponseCache()` returning `ResponseCache` with `getOrFetch(key, fetch)`). Optional `responseCache` in `DatasetToolDeps`; tools build the key and pass a fetch lambda that calls the backend.
- **Line windowing for read operations**: `readDataset` supports `startLine` (1-based, default 1) and `lineCount` parameters. Large files are auto-truncated to `MAX_READ_LINES = 1000` lines when no explicit window is requested. `_result` includes `totalLines`, `startLine`, `returnedLines`, `contentLength`, `mimeType`, and `hasMore`. When `hasMore` is true, the envelope `messages` array contains a directive to call again with the next `startLine`/`lineCount`; the agent must keep calling until `hasMore` is false. Returned text is sanitized: unprintable characters (control chars, DEL, C1 controls) are replaced with `.` for safe JSON and display. The ETag always reflects the full content, not the window. Full reads are cached (same pattern as list tools) so pagination does not re-read from the backend.
- **Data set search**: The `searchInDataset` tool searches for a string in a sequential data set, PDS, or PDS/E (all members or one member). The native backend uses **ZNP `client.tool.search`** when the SDK provides it (PR #809+); otherwise falls back to the shared **search runner** (`src/zos/search-runner.ts`) that calls `listMembers` and `readDataset` and greps in-server. The `NativeToolApi` interface in `native-backend.ts` types the SDK's `tool.search` RPC; `mapZnpSearchResponse()` maps the SDK response to `SearchInDatasetResult`. Detection: `(client as unknown as { tool?: NativeToolApi }).tool?.search`; when present and `ZOWE_MCP_SEARCH_FORCE_FALLBACK` is not set, the ZNP path is used. Results are paginated by member (offset/limit); the full result is cached so paging does not re-run the search. **Natural options** (caseSensitive, cobol, ignoreSequenceNumbers, doNotProcessComments, includeContextLines) are mapped to a SuperC parms string via `buildParmsFromOptions()` in `src/zos/search-options.ts` (e.g. `caseSensitive: false` → ANYC, `doNotProcessComments: ['cobolComment']` → DPCBCMT, `includeContextLines: true` → LPSF). **LPSF context lines**: When `includeContextLines` is true, the tool appends `LPSF` to parms; ZNP tool.search returns `beforeContext` and `afterContext` (±6 lines) per match. These optional `string[]` fields are on `SearchMatchEntry` in `backend.ts` and exposed in the output schema. The fallback path ignores LPSF (no context lines). **Force-fallback**: Set `ZOWE_MCP_SEARCH_FORCE_FALLBACK=1` (or `true`) to force the list+read+grep path even when `tool.search` is available; useful for performance comparison.
- **Encoding (mainframe EBCDIC)**: All data in the MCP server is UTF-8. ZNP uses `localEncoding: 'utf-8'` (constant) for results; ZNP `encoding` is the mainframe (EBCDIC) encoding. **Resolution order** for data set operations: operation param (`encoding`) → per-system override (from `setSystem`) → MCP server default. Defaults: `defaultMainframeMvsEncoding = "IBM-037"` (data sets), `defaultMainframeUssEncoding = "IBM-1047"` (USS). IBM-N/NN is normalized to IBM-NNN (e.g. IBM-37 → IBM-037) for ZNP compatibility. Configurable at startup (CLI `--default-mvs-encoding`/`--default-uss-encoding`, env `ZOWE_MCP_DEFAULT_MVS_ENCODING`/`ZOWE_MCP_DEFAULT_USS_ENCODING`), via VS Code settings (`zoweMCP.defaultMainframeMvsEncoding`, `zoweMCP.defaultMainframeUssEncoding`; changes sent as `encoding-options-update` and take effect immediately), and per-system via `setSystem` optional params (`mainframeMvsEncoding`, `mainframeUssEncoding`; `null` = use server default). Unprintable characters in tool output are replaced with `.` by `sanitizeTextForDisplay()` in `src/tools/response.ts` (single point, applied just before returning read and search results).
- **USS tools**: List/read/write/create/delete/chmod/chown/chtag/copyUssFile, `runSafeUssCommand`, temp tools (`getUssTempDir`, `getUssTempPath`, `createTempUssFile`, `createTempUssDir`, `deleteUssTempUnderDir`), `getUssHome`, and `changeUssDirectory`. **copyUssFile**: Copies USS files or directories; supports `recursive`, `followSymlinks`, `preserveAttributes`, `force` options. Native backend uses `uss.copyUss`; mock uses `fs.cp`. Parameters: `sourcePath`, `targetPath` (absolute or relative to cwd). **USS current working directory (cwd)**: Per-system `SystemContext.ussCwd`; when unset, effective cwd is `ussHome`. Set via `changeUssDirectory`; exposed in `getContext` as `ussCwd`. All USS path-accepting tools accept **absolute** paths (leading `/`) or **relative** paths (resolved against effective cwd). Response paths under cwd are returned in **display form** (relative, forward-only); others are absolute. **listUssFiles** response includes `_context.currentDirectory`, `_context.listedDirectory`, and each entry has a **path** field (relative if under cwd, else absolute). **hardstop-patterns** (npm) enforces evaluation order: for commands, dangerous → block, safe → allow, unknown → elicit or deny; for `readUssFile` path, dangerous → block, sensitive → warn/elicit, safe → allow, unknown → elicit or deny. Single command tool only: `runSafeUssCommand`. USS encoding: same resolution order as datasets; default USS is IBM-1047 (verify ZNP file tag behavior when applicable). `getUssHome` uses `backend.getUssHome()` (which calls `uss.issueCmd('echo $HOME')` on the native backend); result cached in `SystemContext.ussHome` and exposed in `getContext`. Temp delete safety: `deleteUssTempUnderDir` requires path to contain segment `tmp` (or TMP) and at least 3 path segments.
- **Console tools** (**disabled** — ZNP does not yet support `console.issueCmd`; code kept, registration commented out in `server.ts`): `runConsoleCommand` (`src/tools/console/console-tools.ts`) issues z/OS operator console commands; **console-command-patterns.json** defines three outcomes: **block** (system-shutdown: QUIESCE, HALT, Z EOD, SHUTDOWN), **elicit** (SET, VARY, CANCEL, FORCE, STOP, MODIFY, START, ROUTE, unknown commands — require user approval), **safe** (DISPLAY/D commands: D T, D A, D M, D IPLINFO, D R, D J, D SMF, D ASM, D PROG, D PARMLIB). Uses the shared `evaluateCommandSafety()` from `src/tools/command-safety.ts`. Backend: `runConsoleCommand(systemId, commandText, consoleName?, userId?, progress?)`; native uses ZNP `client.console.issueCmd`; mock returns canned output for DISPLAY commands.
- **Restore dataset**: `restoreDataset` (`src/tools/datasets/dataset-tools.ts`) recalls a migrated data set from HSM. Backend: `restoreDataset(systemId, dsn, progress?)`; native uses ZNP `ds.restoreDataset`; mock is a no-op.
- **TSO tools**: `runSafeTsoCommand` (`src/tools/tso/tso-tools.ts`) issues TSO commands; **tso-command-patterns.json** defines three outcomes: **block** (no question — e.g. DELETE/RENAME system data sets SYS1/SYSL/SYS2, PASSWORD, CALL, ALTER, OSHELL), **elicit** (user approval required — e.g. DELETE/RENAME own data set, SUBMIT), **safe** (allow). Evaluation order: dangerous → block, elicit → elicit, safe → allow, unknown → elicit. Uses the shared `evaluateCommandSafety()` from `src/tools/command-safety.ts` via `validateTsoCommand()` in `tso-command-validation.ts`. Output is paginated by line with cached full result; requesting the same command **without** `startLine`/`lineCount` re-executes. Backend: `runTsoCommand(systemId, commandText, userId?, progress?)`; native uses ZNP `client.tso.issueCmd` (SDK 0.3.0+; was `client.cmds.issueTso` in 0.2.x); mock returns canned output for LISTDS, LISTALC, LISTCAT, STATUS, HELP, WHO, TIME, SYSTEM (OSHELL is always blocked; fails with rc 255 in ZNP).
- **Jobs tools**: `submitJob`, `getJobStatus`, `listJobFiles`, `readJobFile`, `getJobOutput`, `searchJobOutput`, `listJobs`, `getJcl`, `cancelJob`, `holdJob`, `releaseJob`, `deleteJob` (`src/tools/jobs/jobs-tools.ts`). **Job card**: JCL requires a JOB statement; when the submitted JCL does not include one, the server prepends the configured job card for the connection. Job cards are loaded from the native config file (`--config`) optional `jobCards` section (key = connection spec `user@host` or `user@host:port`, value = multi-line job card text) and from VS Code setting `zoweMCP.jobCards` (same shape); extension sends `job-cards-update` on connect and on change. Pipe events: `request-job-card` (server→ext), `job-card` (ext→server), `store-job-card` (server→ext). **CLI plugin configuration hot-reload**: `zoweMCP.cliPluginConfiguration` changes send a `cli-plugin-configuration-update` pipe event (ext→server); the server updates `profilesByType` and `activeProfileId` on each named plugin's `CliPluginState` in-place so new profiles take effect immediately without restart. The initial configuration is also sent on every new pipe connection. `zoweMCP.cliPluginConfiguration` is **not** in `affectsServerStartup` (no Cursor server restart on change). **submitJob wait**: When `wait: true` (and optionally `timeoutSeconds`), submitJob waits for the job to reach OUTPUT with adaptive polling (2s initial, +1s per poll up to 15s cap); default timeout 5 minutes; job continues on z/OS after timeout. **Pagination**: `listJobFiles`, `listJobs`, `getJobOutput` (by job file), and `searchJobOutput` (by match) use `offset`/`limit`, `_result.hasMore`, and envelope `messages` to direct the agent to fetch the next page. `readJobFile` uses line windowing (`startLine`/`lineCount`) with `_result.hasMore` and messages. Destructive: `submitJob`, `cancelJob`, `holdJob`, `deleteJob`. Mock backend throws "Not implemented" for all job operations; native uses ZNP `client.jobs.*` (submitJcl, getStatus, listSpools, readSpool, listJobs, getJcl, cancelJob, holdJob, releaseJob, deleteJob).
- **Zowe Explorer open-in-editor**: When Zowe Explorer is available and the extension pipe is connected, the server registers `openDatasetInEditor`, `openUssFileInEditor`, and `openJobInEditor` (`src/tools/zowe-explorer/open-in-editor.ts`). Availability is set at startup via env `ZOWE_EXPLORER_AVAILABLE=1` and **updated at runtime** via the `zowe-explorer-update` pipe event (extension → server): the extension sends current status when the pipe connects and on `vscode.extensions.onDidChange`, so tools appear when Zowe Explorer is installed or activated after the server started. Stdio: tools are registered on the existing server when the event has `available: true`. HTTP: new sessions get the tools when the ref is set. Each tool sends a ServerToExtensionEvent (`open-dataset-in-editor`, etc.) with system/connectionKind for profile resolution. The extension (`event-handler.ts`) uses a generic `handleOpenInEditor()` handler that checks Zowe Explorer availability, resolves the profile, and opens the document; each event type supplies a `buildSpec` callback (returning `OpenInEditorSpec` with `scheme`, `pathPart`, `displayPath`) for URI construction. Profile resolution order: session cache (per system key) → default from ProfileInfo → default from `zowe config list --rfj` (with workspace cwd for project-local zowe.config.json) → match by system (host/user) → quick pick of zosmf profiles or input box to type profile name. Resolved or chosen profile is stored in session for the system so the next open is immediate. URIs: zowe-ds (path segments encoded so `#` in member names works), zowe-uss (path segments encoded), zowe-jobs (jobId or jobId/jobFileId). Documents open with `preview: false` so multiple opens do not replace the previous tab. Profile listing uses `getAllZosmfProfileNames(workspaceDir)` (ProfileInfo first, then CLI); `getZosmfProfilesFromZoweCli(workspaceDir)` returns `{ names, defaultName }` from `zowe config list --rfj`. Extension activates Zowe Explorer when building server definitions so it is ready when opening resources.
- **Dual MCP registration (VS Code + Cursor)**: The extension registers the Zowe MCP server in two ways. (1) **VS Code**: `mcpServerDefinitionProviders` in `package.json` and `vscode.lm.registerMcpServerDefinitionProvider()` — the standard API used by VS Code (and documented in the VS Code MCP guide). (2) **Cursor**: when running in Cursor, the extension also calls `vscode.cursor?.mcp?.registerServer()` and `vscode.cursor?.mcp?.unregisterServer('zowe')`. The `vscode.cursor` namespace is Cursor-specific and not part of the official VS Code API; it is not formally documented. Always guard with `typeof vscode.cursor?.mcp?.registerServer === 'function'` so the extension runs in plain VS Code without errors. On settings change, unregister then re-register with Cursor so the server restarts with updated config. On deactivate, unregister if we had registered.

## Common Patterns

### Adding a New Tool

1. Create a file under `packages/zowe-mcp-server/src/tools/<component>/`
2. Export a `register<Component>Tools(server: McpServer, deps, logger)` function
3. Import and call it from `packages/zowe-mcp-server/src/server.ts`
4. **Tool names**: Use **camelCase** (e.g. `listDatasets`, `setSystem`) so they fit VS Code Copilot; avoid snake_case.
5. For data set tools, accept `DatasetToolDeps` (backend, systemRegistry, sessionState, credentialProvider, encodingOptions, optional responseCache). For context tools, accept `ContextToolDeps` (serverVersion, backendKind, optional systemRegistry/sessionState/credentialProvider).
6. Use `readOnlyHint: true` annotation for read-only tools, `destructiveHint: true` for destructive ones.
7. Add progress: pass `extra` (second handler arg), build a title from args (e.g. "List members of " + dsn), call `createToolProgress(extra, title)`, then `await progress.start()` at start and `await progress.complete(statusOrError)` in success/error paths. Optionally pass `progress.step` to the backend when `extra._meta?.progressToken` is set.
8. **Tool description**: Start with a clear, complete first sentence (this appears in the tools table). Do not start with use-case phrases ("For automation"), "Destructive.", or include "e.g." in the first sentence. For paginated tools, use `withPaginationNote(description, PAGINATION_NOTE_LIST)` or `withPaginationNote(description, PAGINATION_NOTE_LINES)` from `response.ts` — do not inline the pagination protocol.
9. **Output schema**: Define a Zod schema in the component's `*-output-schemas.ts` (or create that file following `dataset-output-schemas.ts`). Use the scoped context schema for the envelope (`baseContextSchema` for jobs/TSO/console, `datasetContextSchema` for datasets, `ussContextSchema` for USS). Export a per-tool schema and set `outputSchema` on `registerTool`. Success responses must return the shape declared (use `wrapResponse()` which adds `structuredContent`). For error paths (e.g. validation, blocked action), return `{ content: [{ type: 'text', text: JSON.stringify({ error: message }) }], isError: true }` so clients do not validate the error body against the success schema.
10. Add tests in `packages/zowe-mcp-server/__tests__/`

### Adding a New Component

1. Create a new directory under `packages/zowe-mcp-server/src/tools/<component>/`
2. Follow the pattern in `src/tools/datasets/dataset-tools.ts` (for z/OS tools) or `src/tools/context/context-tools.ts` (for always-available tools)
3. Register in `server.ts` — z/OS tools go inside the `if (hasBackend)` block
4. Update the `components` array in the `getContext` tool handler (`context-tools.ts`)

### Command Safety Evaluation

Pattern-based command validation for TSO and console tools uses the shared `evaluateCommandSafety()` function from `src/tools/command-safety.ts`. Each component provides its own JSON patterns file (`tso-command-patterns.json`, `console-command-patterns.json`) with `dangerous`, `elicit`, and `safe` arrays. Pattern entries have `id`, optional `message`, and `pattern` (regex). Evaluation order: dangerous → block, elicit → elicit, safe → allow, unknown → elicit. The TSO wrapper is `validateTsoCommand()` in `src/tools/tso/tso-command-validation.ts`; console calls `evaluateCommandSafety()` directly. To add a new command safety domain, create a patterns JSON file with the `CommandPatterns` shape and call `evaluateCommandSafety()`.

### Response Envelope for Dataset Tools

All data set tool responses use the envelope from `src/tools/response.ts`:

- **Types**: `ToolResponseEnvelope<T>`, `ResponseContext`, `ListResultMeta`, `ReadResultMeta`, `MutationResultMeta`, `SearchResultMeta`
- **Helpers**: `buildContext()`, `formatResolved()`, `resolvedOnlyIfDifferent()`, `paginateList()`, `getListMessages()`, `paginateSearchResult()`, `windowContent()`, `getReadMessages()`, `sanitizeTextForDisplay()`, `wrapResponse()`
- **Constants**: `DEFAULT_LIST_LIMIT` (500), `MAX_LIST_LIMIT` (1000), `MAX_READ_LINES` (1000)

When adding a new data set tool:

1. Use `buildContext(systemId, { resolvedDsn: resolvedOnlyIfDifferent(resolved, rawInput) })` so resolved keys are only present when resolution changed the input (or `resolvedPattern` for list, `resolvedTargetDsn` for copy/rename)
2. Use `paginateList()` for list tools (pass `getListMessages(meta)` as fourth arg to `wrapResponse`); use `sanitizeTextForDisplay()` then `windowContent()` for read tools (pass `getReadMessages(meta)` as fourth arg to `wrapResponse` when there are more lines)
3. Use `wrapResponse(ctx, meta, data, messages)` to assemble the final MCP response
4. For mutation tools, use `MutationResultMeta` with `{ success: true }`

The `resolvedPattern` field is for list tools (`listDatasets`); `resolvedDsn` is for CRUD tools; `resolvedTargetDsn` is for copy/rename operations.

### MCP output schemas (Zod)

Tools that return structured data declare an `outputSchema` (Zod) so the MCP server advertises the response shape in `tools/list` and can return `structuredContent` that clients validate.

- **Location**: Per-component files `src/tools/<component>/*-output-schemas.ts` (e.g. `dataset-output-schemas.ts`, `uss-output-schemas.ts`, `jobs-output-schemas.ts`). Shared envelope pieces (e.g. `baseContextSchema`, `listResultMetaSchema`) live in `dataset-output-schemas.ts` and are imported by USS/TSO/jobs.
- **Scoped context schemas**: The `_context` Zod schema is scoped per component so each tool group only advertises the fields it uses: `baseContextSchema` (`{ system }`) for jobs, TSO, and console; `datasetContextSchema` (`{ system, resolvedPattern?, resolvedDsn?, resolvedTargetDsn? }`) for dataset tools; `ussContextSchema` (`{ system, resolvedPath?, currentDirectory?, listedDirectory? }`) for USS tools. The deprecated `responseContextSchema` (full union) is kept for backward compatibility but should not be used in new code.
- **Success responses**: Use `wrapResponse(ctx, meta, data, messages)` which returns both `content` (JSON string) and `structuredContent` (the envelope). The Zod schema must describe that envelope (e.g. `_context`, `messages`, `data`, optional `_result`).
- **Error responses**: Return `{ content: [{ type: 'text', text: JSON.stringify({ error: message }) }], isError: true }`. Do not include `structuredContent`; the MCP spec does not require tool execution errors to conform to the success output schema.
- **Components with output schemas**: context (getContext — includes server info; listSystems, setSystem), datasets (all 14), USS (all 16), TSO (runSafeTsoCommand), jobs (all 15). **Without output schema**: Zowe Explorer open-in-editor tools (openDatasetInEditor, openUssFileInEditor, openJobInEditor).

### z/OS Domain Model

- **System identity**: `SystemId` (hostname string) in `src/zos/system.ts`. `SystemRegistry` manages known systems.
- **Credentials**: `CredentialProvider` interface in `src/zos/credentials.ts`. Implementations: `MockCredentialProvider` (reads `systems.json`), future: `ZoweTeamConfigProvider`, `OAuthTokenProvider`.
- **Session state**: `SessionState` in `src/zos/session.ts`. Tracks active system and per-system `SystemContext` (userId).
- **DSN utilities**: `src/zos/dsn.ts` — `resolveDsn(input, member?)`, `resolvePattern(input)`, `validateDsn()`, `validateMember()`, `buildDsUri()`, `inferMimeType()`. Use `dsn` (not `dsname`) as the variable name for data set names throughout the codebase.
- **Backend**: `ZosBackend` interface in `src/zos/backend.ts`. Implementations: `FilesystemMockBackend` (mock), `NativeBackend` (Zowe Native Proto SDK over SSH; listDatasets, listMembers, readDataset, searchInDataset). Search is implemented via `runSearchWithListAndRead()` in `src/zos/search-runner.ts` (listMembers + readDataset + grep); when ZNP `tool.search` is available, Native can switch to it.
- **Temporary data sets**: Temp prefix/name generation and cleanup live in `src/zos/temp-dsn.ts`. Default HLQ is `userId.TMP`. Prefix and DSN are verified unique on the system (list with `prefix.**` for prefix, `getAttributes` for single DSN; retry with new candidate if in use). Tools: `getTempDatasetPrefix`, `getTempDatasetName`, `createTempDataset`, `deleteDatasetsUnderPrefix`. Cleanup requires at least 3 qualifiers and a `TMP` qualifier (e.g. `USER.TMP.XXXXXXXX.YYYYYYYY`); list with `prefix.**` then delete each (deepest first). Works with both mock and native backends.

### Mock Mode

- **Starting**: `npx @zowe/mcp-server --stdio --mock ./zowe-mcp-mock-data` or `ZOWE_MCP_MOCK_DIR=./zowe-mcp-mock-data`
- **Generating data**: `npx @zowe/mcp-server init-mock --output ./zowe-mcp-mock-data [--preset minimal|default|large|inventory|pagination]`. Use **inventory** preset for one data set (USER.INVNTORY) with 2000 members for listMembers pagination E2E. Use **pagination** preset for both inventory (2000 members) and USER.PEOPLE.firstname.lastname (1000 PS data sets) for listMembers and listDatasets evals. Optional: `--inventory-members N`, `--people-datasets N`, `--seed N` (default 42) for reproducible Faker data. People names are unique, English, no special characters, first and last each ≤8 chars.
- **Directory layout**: DSFS-inspired — `zowe-mcp-mock-data/{system}/{HLQ}/{dataset-qualifiers}/` with `_meta.json` for attributes
- **Adding mock systems**: Edit `systems.json` in the mock data directory. Each system has `host`, `port`, `description`, optional `defaultUser`, and `credentials` array. If `defaultUser` is omitted, the first credential entry is used as the default.

### Code Formatting and License Headers

- **Auto-formatted by Cursor hook**: `.cursor/hooks/format.sh` runs automatically after every Agent and Tab file edit. For `.ts` files it first runs ESLint `--fix` (to insert the license header if missing), then Prettier. JS/JSON files get Prettier only. Markdown files get markdownlint-cli2. No manual formatting needed during development.
- **License header**: Every `.ts` file must begin with the EPL-2.0 block comment. ESLint (`eslint-plugin-headers`) enforces this via `eslint.config.mjs` and auto-fixes with `--fix`. You do not need to manually add the header — the hook does it for you.

### Code Duplication Detection

- **Tool**: `jscpd` (npm devDependency) — token-based copy/paste detector using the Rabin-Karp algorithm. Detects exact and near-exact duplicated code blocks across TypeScript and JavaScript files.
- **Config**: `.jscpd.json` at repo root. Key settings: `minLines: 10` (minimum block size), `minTokens: 50`, `threshold: 5` (fail if >5% duplication). Excludes `__tests__/`, `themes/`, `dist/`, `out/`, `server/`, `.vscode-test/`, `*.d.ts`.
- **Manual scan**: Run `npm run duplication` to scan all packages and report duplicates to console.
- **Cursor hook**: `.cursor/hooks/duplication.sh` runs automatically after Agent file edits (not Tab edits) on `.ts` files under `packages/`. It runs jscpd on the edited file and reports any duplication to the agent as advisory output. Always exits 0 (non-blocking).
- **CI integration**: jscpd exits non-zero when duplication exceeds the configured `threshold`, suitable for CI quality gates.

### Logging (MCP Server)

- **Logger class**: `packages/zowe-mcp-server/src/log.ts` exports a `Logger` class with triple output: stderr (always), MCP protocol notifications (when connected), and VS Code extension pipe (when connected). Created as a singleton via `getLogger()` in `server.ts`.
- **Log levels**: RFC 5424 syslog levels — `debug`, `info`, `notice`, `warning`, `error`, `critical`, `alert`, `emergency`. Default is `info`, overridden by the `ZOWE_MCP_LOG_LEVEL` environment variable or dynamically via `log-level` events from the VS Code extension.
- **Child loggers**: Use `logger.child('name')` to create a named child logger that shares the parent's server attachment, extension client, and level. Each component/transport should create its own child (e.g., `logger.child('http')`, `logger.child('core')`).
- **Usage in tools**: Tool registration functions receive a `Logger` parameter from `server.ts`. Create a child logger and use it for diagnostic messages (e.g., `log.debug('tool called')`).
- **Usage in transports**: Transport functions (`startStdio`, `startHttp`) receive a `Logger` parameter. Create a child logger named after the transport.
- **Do not use `console.log` or `console.error`**: Always use the `Logger` class. For stdio transport, writing to stdout corrupts the JSON-RPC protocol.
- **Stderr format**: `YYYY-MM-DDTHH:mm:ss.sssZ [LEVEL] [name] message {data}`

### Extension Communication (Named Pipe)

- **Pipe server**: `packages/zowe-mcp-vscode/src/pipe-server.ts` creates a `net.Server` on a per-workspace named pipe. The pipe path is `/tmp/zowe-mcp-<workspaceId>.sock` (Unix) or `\\.\pipe\zowe-mcp-<workspaceId>` (Windows). A discovery JSON file is written to `context.globalStorageUri`.
- **Extension client**: `packages/zowe-mcp-server/src/extension-client.ts` reads the discovery file and connects to the pipe. Uses retry logic (up to 10 attempts, 1s delay). Returns `undefined` in standalone mode (env vars absent).
- **Event types**: Defined in `packages/zowe-mcp-server/src/events.ts`. The `McpEvent<T, D>` generic envelope carries a `type`, `data`, and `timestamp`. `ServerToExtensionEvent` includes `log`, `notification`, `request-password`, `password-invalid`, `store-password`, `request-job-card`, `store-job-card`, `open-dataset-in-editor`, `open-uss-file-in-editor`, `open-job-in-editor`, `ceedump-collected`; `ExtensionToServerEvent` includes `log-level`, `password`, `systems-update`, and others.
- **Event handler**: `packages/zowe-mcp-vscode/src/event-handler.ts` dispatches incoming server events to VS Code APIs (e.g. `log` events → `LogOutputChannel`, `notification` events → `vscode.window.showWarningMessage` / `showErrorMessage` / `showInformationMessage` with an "Open Settings" button). The `ceedump-collected` event includes `path`, `reason` (abend text, e.g. CEE3204S), `znpOperation`, and `mcpTool`; the extension shows an information message with that context and an "Open Dump" button that opens the file in the editor, and logs a line to the Zowe MCP output channel with the full dump path. Open-in-editor events (`open-dataset-in-editor`, `open-uss-file-in-editor`, `open-job-in-editor`) resolve the Zowe profile, build the appropriate URI (zowe-ds, zowe-uss, zowe-jobs), and open the document with `preview: false` so multiple opens keep separate tabs.
- **Adding a new event type**: (1) Add the event data interface and type alias to `events.ts`. (2) Add it to the appropriate union (`ServerToExtensionEvent` or `ExtensionToServerEvent`). (3) Handle it in `event-handler.ts` (server→extension) or register a handler via `extensionClient.onEvent()` (extension→server).
- **Env vars**: `MCP_DISCOVERY_DIR` and `WORKSPACE_ID` are passed to the MCP server via `McpStdioServerDefinition`'s `env` parameter. They are set automatically by the extension — no manual configuration needed.

## Naming Conventions and Glossary

- **"data set" (two words)**: All user-facing strings (tool descriptions, `.describe()`, prompt text, resource descriptions, error messages) use "data set" (two words), matching IBM documentation. Code identifiers use "dataset" (one word, e.g. `readDataset`, `DatasetEntry`).
- **PDS and PDS/E only**: The correct names are **PDS** (Partitioned Data Set) and **PDS/E** (Partitioned Data Set Extended). "PDSE" is a common typo (missing `/`) — it is accepted as a valid input alias (in `schema-utils.ts` and code logic) but never used in descriptions, comments, or documentation.
- **"PDS or PDS/E"**: When referring to both types, write "PDS or PDS/E". The slash in PDS/E is part of the name (like TCP/IP), not an "or" separator, so "PDS/E" alone means only the extended format.
- **"PDS/E (recommended)"**: In `createDataset`/`createTempDataset` type descriptions, PDS/E is marked as the recommended choice for new data sets.
- **Accepted input aliases**: The `datasetTypeSchema` in `schema-utils.ts` accepts `PO-E`, `PDSE`, and `LIBRARY` as case-insensitive aliases for PDS/E. These remain valid input values.
- **PO-E**: The z/OS DSORG value for PDS/E. Used as-is when describing system values (e.g. "DSORG=PO-E"), with parenthetical `(PDS/E)` for clarity.
- **z/OS glossary in SERVER_INSTRUCTIONS**: A glossary of z/OS terminology is appended to `SERVER_INSTRUCTIONS` in `server.ts` so every LLM client gets canonical definitions (data set, PDS, PDS/E, member, sequential, DSN, USS, TSO, JCL, RECFM, LRECL, VSAM, HSM, EBCDIC, HLQ).

The full pagination protocol is in `SERVER_INSTRUCTIONS` (sent to clients at init); tool descriptions carry only a short reference via `withPaginationNote()`.
- Parameter descriptions for data set/pattern inputs document fully qualified names (e.g. USER.SRC.COBOL).

- Mock ETags are derived from file modification timestamps (`mtime`). Real backends should use the ETag mechanism provided by their API (e.g. z/OSMF `ETag` header).
- The `init-mock` CLI supports presets (`minimal`, `default`, `large`, `inventory`, `pagination`) and custom scale parameters (`--systems`, `--users-per-system`, `--datasets-per-user`, `--members-per-pds`, `--inventory-members N`, `--people-datasets N`, `--seed N` default 42 for reproducible inventory/Faker data). People data sets are USER.PEOPLE.firstname.lastname (PS); names use the same seed as inventory.
- **Server-side member pattern filtering**: When the native backend calls ZNP `listDsMembers`, it passes the `pattern` parameter (converting MCP `%` to ZNP `?`, `*` passes as-is). Client-side filtering via `memberPatternToRegExp` is still applied as a fallback to handle edge cases. This reduces data transfer for large PDS libraries.
- **Richer data set attributes**: `DatasetEntry` and `DatasetAttributes` include: `referenceDate`, `expirationDate`, `multivolume`, `migrated`, `encrypted`, `dsntype`, `dataclass`, `mgmtclass`, `storclass`, `spaceUnits`, `usedPercent`, `usedExtents`, `primary`, `secondary`, `devtype`, `volsers`. The native backend maps all available SDK fields via `mapDatasetToEntry`. Output schemas (`datasetListEntrySchema`, `getDatasetAttributesDataSchema`) include all fields. The `migrated` field is particularly useful for the LLM to decide whether to call `restoreDataset`.
- **listDatasets detail levels**: The `listDatasets` tool has a `detail` parameter (`minimal`, `basic`, `full`; default `basic`) that controls response verbosity. The backend always fetches full attributes; filtering happens in the tool layer via `filterDatasetFields()` in `dataset-tools.ts`. At non-full levels, `suppressDefaults()` strips `migrated: false`, `encrypted: false`, and single-element `volsers` (noise reduction). **minimal**: `dsn`, `dsorg`, `dsntype`, `migrated`, `encrypted` (booleans only when true); `volser` only for non-SMS/non-VSAM. **basic**: adds `recfm`, `lrecl`, `blksz`, `spaceUnits`, `primary`, `secondary`; `volser` only for non-SMS/non-VSAM (no `volsers`). Does NOT include dates, `multivolume`, `usedPercent`, `usedExtents`. `volsers` is full-only. **full**: all fields including `resourceLink`, SMS classes, `devtype`, all dates. The `attributes` boolean parameter was removed; the cache key no longer includes it.
- **VSAM pseudo-volser cleanup**: z/OS returns `*VSAM*` as the volser for VSAM data sets. The `listDatasets` enrichment step strips this pseudo-volser (removes `volser: "*VSAM*"` and filters `*VSAM*` from `volsers`). VSAM data sets are identified by `dsorg: "VS"`. VSAM is always SMS-managed, so in `minimal` detail the filter treats `dsorg === 'VS'` the same as having a `storclass` (no volser shown). The `migrated` field defaults to `false` when the backend does not set it.
- **Native backend list pattern**: The Zowe Native Proto z/OS server appends `.**` to list patterns that do not end with a wildcard (so e.g. `USER` becomes `USER.**`). Invalid patterns like `'...'` (empty qualifiers) would become `'....**'` and produce confusing errors; `validateListPattern()` in `dsn.ts` rejects empty qualifiers before calling the backend so the user gets a clear error.
- **writeDataset block of records**: When the tool passes optional `startLine` and `endLine` (1-based, inclusive), the backend replaces that line range with the given content; the number of lines in content need not match (data set can grow or shrink). When only `startLine` is provided, the same number of lines as in content are replaced. Both mock and native implement this as read-modify-write; native uses the ETag from the read for the subsequent write.
- **Native copyDataset**: The Zowe Native Proto SDK has no copy RPC; the native backend implements `copyDataset` as `readDataset` then `writeDataset`. The target data set must already exist.
- **ZNP SDK 0.3.0 API restructuring**: The `cmds` API group was split in SDK 0.3.0: `client.cmds.issueUnix` → `client.uss.issueCmd`, `client.cmds.issueTso` → `client.tso.issueCmd`. New API: `client.tool.search` (SuperC search on z/OS). The `ds`, `uss` (file ops), and `jobs` groups are unchanged. The native backend (`native-backend.ts`) uses `NativeTsoApi`/`getTso()` for TSO, `NativeUssApi.issueCmd` for Unix commands, and `NativeToolApi`/`getTool()` for search.
- **Auto-redeploy outdated ZNP server**: `SshClientCache.getOrCreate()` compares the remote z/OS server checksums (`client.serverChecksums`) against the local SDK checksums (`ZSshUtils.checkIfOutdated`) after every successful connection. When a mismatch is detected and `autoInstallZnp` is true, the cache disposes the old client, redeploys via `ZSshUtils.installServer()`, and reconnects with a doubled response timeout. This ensures the z/OS `zowed` binary stays in sync with the Node.js SDK (e.g. after switching to a newer SDK via `npm run sdk:pr`), preventing "Unrecognized command" errors from version mismatches.
- **ZNP server-not-found detection**: `isZnpServerNotFoundError()` in `ssh-client-cache.ts` matches three error patterns: (1) `"Server not found"` — the SDK's specific error when FSUM7351 is in the shell output, (2) `"FSUM7351"` — USS shell "not found" code, (3) `"Error starting Zowe server"` — the SDK's generic fallback when the binary cannot start but the shell error is not FSUM7351 (e.g. different locale, /bin/sh "not found" message). All three trigger auto-install when `autoInstallZnp` is true.
- **ZNP OSHELL**: TSO command "OSHELL cmd" fails with rc 255 in ZNP; do not use it for home resolution or other USS discovery.
- **SDK switching**: Seven modes for acquiring the Zowe Native Proto SDK (`scripts/sdk-switch.js`): (0) `npm run sdk:fallback` — uses the committed `resources/zowe-native-proto-sdk-<version>.tgz`; used by CI by default; (1) `npm run sdk:release [-- version]` — latest or specific release from Zowe Artifactory npm; (2) `npm run sdk:nightly` — latest nightly from Artifactory `libs-snapshot-local` (falls back to GitHub Actions Build on `main`); (3) `npm run sdk:pr -- <pr-number>` — artifact from a PR's Build workflow; (4) `npm run sdk:branch -- <branch>` — latest successful Build run for a branch; (5) `npm run sdk:local -- <path>` — local `.tgz` file or `zowe-native-proto` repo directory (auto-detects; if directory, runs `npm pack` in `packages/sdk`). All modes download/copy the tarball to `resources/zowe-native-proto-sdk-<version>.tgz` and set the server's `package.json` dependency to `file:../../resources/zowe-native-proto-sdk-<version>.tgz`. The script runs `npm install` after switching. After switching, rebuild (`npm run build`) and run tests (`npm test`) to verify compatibility.
- **ZNP tool.search detection**: In `NativeBackend.searchInDataset`, the code checks `(client as unknown as { tool?: NativeToolApi }).tool?.search`. When present (SDK 0.3.0+ from PR #809), it calls `tool.search({ dsname, string, parms })` and maps the response via `mapZnpSearchResponse()`. When absent (SDK 0.2.x), or when `ZOWE_MCP_SEARCH_FORCE_FALLBACK=1`, the fallback list+read+grep path is used. The `NativeToolApi` interface in `native-backend.ts` types the SDK's `tool.search` RPC response shape including optional `beforeContext`/`afterContext` per match.
- **Search benchmark**: `__tests__/search-benchmark.test.ts` compares ZNP tool.search vs fallback against real z/OS. Requires `ZOWE_MCP_SEARCH_BENCHMARK=1`, `native-config.json`, and password. Customize DSN/string with `ZOWE_MCP_SEARCH_BENCHMARK_DSN` (default `SYS1.PARMLIB`) and `ZOWE_MCP_SEARCH_BENCHMARK_STRING` (default `SYSTEM`). Run: `ZOWE_MCP_SEARCH_BENCHMARK=1 npx vitest run search-benchmark --testTimeout=300000` from the server package. Reports wall-clock time, lines found/processed, and member counts for both paths.
- **build-and-install**: The extension's `bundle:server` step (`scripts/bundle-server.js`) copies the server dist into an isolated `server/` directory and runs `npm install --omit=dev` there. Workspace dependencies (e.g. `zowe-mcp-common`) are not published to any registry; the script bundles them locally into `server/.local/<name>` and rewrites the dep to `file:.local/<name>`. After install, symlinks created by `file:` deps are replaced with real directory copies (vsce/yazl cannot pack symlinks into a VSIX). The deps cache hash includes the common package's `package.json` so it invalidates when common changes. When using a private npm registry (e.g. Artifactory), ensure `npm login` has been run so the bundle install can authenticate (E401 otherwise).
- **Open-in-editor URIs**: Build zowe-ds/zowe-uss/zowe-jobs URIs with path segments percent-encoded (`encodeURIComponent` per segment) so member names like `###` work; open documents with `preview: false` so each open gets its own tab.
- **Output schema and errors**: When a tool has `outputSchema`, success results must include `structuredContent` that conforms. Return `isError: true` for tool execution errors (blocked command, validation failure, user decline) so the client does not validate the error body against the success schema; the `content` array still carries the JSON error message for the LLM.
- **Zowe CLI Plugin Bridge**: A fully generic, metadata-driven mechanism (`src/tools/cli-bridge/`) that creates MCP tools from Zowe CLI plugin commands defined in YAML. No plugin-specific code lives in the TypeScript source — all plugin-specific details are declared in the YAML. Each tool invokes `zowe ... --rfj` via `spawnSync` and wraps the JSON output in the standard MCP response format. Key types: `ProfileFieldDef` (field in a profile type), `ProfileTypeDef` (profile type with `name`, `toolListName`, `toolSetName`, `required`, `perToolOverride`, `fields`), `CliNamedProfile` (`{ id, ...fieldValues }`), `CliPluginProfilesFile` (`Record<typeKey, { profiles: CliNamedProfile[], default?: string }>`), `CliPluginState` (`profilesByType`, `activeProfileId`, `virtualContextByType`, `passwordResolver`, `configSource`). The `configSource` field (`'vscode' | 'cli'`) is set in `buildPluginState()` based on whether the VS Code extension pipe is connected; it drives context-aware remediation hints in CLI invocation error messages (`buildRemediationHint()` in `cli-tool-loader.ts`). The `sendNotification` callback (set only in VS Code mode) sends a `notification` pipe event so the user sees a native VS Code error dialog on CLI failure. **Non-retryable error pattern**: when a CLI tool call fails, the error payload includes `"stop": true` and a `"suggestion"` field with explicit "do NOT retry" language; `SERVER_INSTRUCTIONS` instructs the LLM to stop and present the suggestion verbatim whenever it sees `stop: true`. The YAML `profiles:` block defines named profile types; `required: true` means a profile must be active before any tool call (auto-selected when only one exists); `perToolOverride: true` means individual fields are injectable per tool call (location-style). Four profile management tools are auto-generated from the YAML `profiles:` block (e.g. `endevorListConnections`, `endevorSetConnection`, `endevorListLocations`, `endevorSetLocation`). The Endevor plugin provides 10 command tools + 4 profile management tools. **Plugin definitions directory**: Plugin YAML files (and their companion `*-commands.json`) live in `src/tools/cli-bridge/plugins/` and are copied to `dist/tools/cli-bridge/plugins/` by `copy-resources.cjs`. **Auto-discovery (default)**: On startup the server scans `<server-dist>/tools/cli-bridge/plugins/` for `*.yaml` files and loads them all. Override the directory with `--cli-plugins-dir <path>` or `ZOWE_MCP_CLI_PLUGINS_DIR` env var (useful for tests that point to the source `src/tools/cli-bridge/plugins/` directory). Select a subset with `--cli-plugin-enable <name>` (repeatable). Supply a profiles config file per plugin with `--cli-plugin-connection <name>=<connfile>` (e.g. `endevor=/tmp/conn.json`). `--cli-plugin-desc-variant <name>` sets `ZOWE_MCP_CLI_DESC_VARIANT`. **VS Code**: `zoweMCP.enabledCliPlugins` (array of plugin names; empty = all) and `zoweMCP.cliPluginConfiguration` (object: plugin name → inline `CliPluginProfilesFile` object; the extension writes a temp file to `globalStorageUri` and passes it to the server). Passwords are NOT stored in the profiles file; set `ZOWE_MCP_PASSWORD_<USER>_<HOST>` env vars. **VS Code (legacy/explicit)**: `zoweMCP.cliPlugins` (array of `{ yaml, connectionFile, descVariant }`) for explicit YAML path overrides — still supported, additive.
- **CLI plugin description variants**: The YAML supports multiple description variants per tool: `cli` (original CLI help text, may use `$.path` JSON reference), `intent` (action-centric rewrite), `optimized` (LLM-tuned). Any custom variant name is valid. Selected by `activeDescription` in the YAML, `ZOWE_MCP_CLI_DESC_VARIANT` env var, or `--cli-plugin-desc-variant` flag (priority: tool > plugin > env > 'intent' > 'cli'). Use `scripts/rephrase-tool-descriptions.mjs` to generate `optimized` variants via Gemini — the script validates completeness (must end with `.!?`), retries incomplete outputs, and strips markdown. Config from `evals.config.json`. Run: `node scripts/rephrase-tool-descriptions.mjs [--yaml <path>] [--model <id>] [--variant optimized] [--source intent]`.
- **`$.path` JSON reference syntax in plugin YAML**: Any string value in a plugin YAML that starts with `$.` is treated as a dotted-path reference into a companion `<plugin>-commands.json` file located next to the YAML. Example: `cli: "$.endevor.list.environments.description"` resolves to the value at that path in `endevor-commands.json`. Implemented in `loadPluginYaml()` via `resolveJsonRefs()` and `resolveJsonRef(path, jsonData)`. Applies to all `descriptions.*` variants, tool `params[].description`, and `profiles.<type>.fields[].description`. When the companion JSON is absent or the path is not found, the original `$.` string is left unchanged. Use this to reference authoritative CLI descriptions from the `endevor-commands.json` file (generated from the Ansible Collection Generator) without duplicating text in the YAML.
- **Profile field `default` in plugin YAML**: A `ProfileFieldDef` may have `default?: string`. When present, `buildCliArgs()` injects the default value even when the model omits the parameter and the virtual context is empty. Example: the `type` location field has `default: "*"` so the Endevor CLI always receives `--typ *` when the model doesn't specify a type. The `stageNumber` field similarly defaults to `"*"`.
- **`MockServerDef`** in `packages/zowe-mcp-evals/src/types.ts`: Defines a generic mock server started by the eval harness. Fields: `name`, `cliScript`, `initArgs?` (run once in a harness-managed temp dir with `{dataDir}` substitution), `configTemplate?` (written to temp file with `{dataDir}`/`{port}` substitution), `configOutputName?`, `configFlag?`, `serveArgs?` (full arg string replacing `serve --port <port>`; use `${availablePort}` for dynamic port allocation starting from 10000 — recommended to avoid port conflicts when running tests in parallel), `startArgs?` (legacy, ignored when `serveArgs` is set), `port?` (ignored when `serveArgs` has `${availablePort}`). Used in question-set YAML as `config.mockServers` (array). The harness creates a temp directory, runs the init command, starts the server, waits for port readiness, and cleans up on stop. **Dynamic port in `mcpServerArgs`**: Use `${port:<serverName>}` in `mcpServerArgs` to reference the resolved port of a named mock server (e.g. `--ews-url=http://localhost:${port:endevor-ews}` after defining a mock server named `endevor-ews` with `serveArgs: "serve --port ${availablePort}"`). **Port allocation**: `findAvailablePort(startFrom=10000)` in `harness.ts` uses a module-level `allocatedPorts` set so multiple harness instances in the same process never pick the same port. The e2e test (`mock-ews-stdio.e2e.test.ts`) has its own local `findAvailablePort` for the same reason. **`CliPluginConnection`** in `types.ts`: Specifies connection details for one CLI plugin (host, port, user, password, protocol, basePath, pluginParams). Used in question-set YAML as `config.cliPluginConfiguration` (map of plugin name to connection). For the `endevor` plugin, the harness fills in defaults (localhost/USER/PASSWORD/http/EndevorService/api/v2/instance=ENDEVOR) so only `port` is usually needed. **CLI bridge set-level config**: `config.cliPluginsDir` (--cli-plugins-dir), `config.cliPluginDescVariant` (--cli-plugin-desc-variant), `config.mcpServerScript`/`mcpServerArgs` (alternative MCP server, e.g. code4z-gen-ai), `config.toolAliases` (normalize assertion tool names).
- **Endevor eval sets**: Standard: `endevor.yaml` (intent variant, 10 q, 5 reps), `endevor-cli.yaml` (cli variant, same q via `questionsFrom`), `endevor-optimized.yaml` (optimized variant, same q via `questionsFrom`), `endevor-code4z.yaml` (code4z-gen-ai, same q via `questionsFrom`). Stress: `endevor-stress.yaml` (intent), `endevor-stress-cli.yaml`, `endevor-stress-optimized.yaml`, `endevor-stress-code4z.yaml`. For variant comparison: `npm run eval-compare -- --set endevor,endevor-cli,endevor-optimized,endevor-code4z,endevor-stress,endevor-stress-cli,endevor-stress-optimized,endevor-stress-code4z --model gemini-2.5-flash --label "label"`. See `docs/endevor-variants-gemini25flash-analysis.md` for full variant comparison results. Key findings: cli=optimized=100% and intent=98% on standard q; optimized=70% > cli=66% > intent=58% on stress q; all Zowe MCP variants tie or beat code4z-gen-ai (90%/70%).
- **`questionsFrom` in question-set YAML**: Set `config.questionsFrom: <set-name>` to load questions from another named set instead of defining them locally. Used by mirror sets (e.g. `endevor-code4z.yaml`) that run the same questions against a different server — avoids duplication and keeps question counts equal for apples-to-apples comparison. When `questionsFrom` is present, the local `questions` array is not required.
- **Eval metrics** (`packages/zowe-mcp-evals`): Each run result captures `durationMs` (wall-clock time), `tokenUsage` (`{ input, output, total }`), and `stepCount` (number of agent reasoning steps). Reported in the Markdown report under "Metrics per question" (avg duration, avg tokens, avg steps). These metrics are populated by the harness from the Vercel AI SDK `result.usage.inputTokens`/`outputTokens` and `result.steps.length`.
- **Eval metrics**: `RunResult` now includes `durationMs` (wall-clock ms), `tokenUsage` (`{ input, output, total }`), and `stepCount` (number of agent steps). These are captured by `harness.runOne()` from the Vercel AI SDK `result.usage` and `result.steps`. The report shows a **"Metrics per question"** table when any run has metrics. Cached runs do not have metrics (metrics only come from live LLM calls).

- **Native backend abend (CEE3204S/0C4)**: When the ZNP server on z/OS abends (e.g. protection exception 0C4), the native backend treats the error as a connection error: it evicts the SSH client immediately (connection closed), rethrows so the client gets the error, and does not reuse that connection. When an abend is detected (invalid JSON containing CEE3204S/0C4/protection exception), the server logs a dedicated **notice** that includes which Zowe Native operation and which MCP tool were in progress (e.g. "ZNP server on z/OS abended during Zowe Native operation 'listDatasets' (MCP tool: 'listDatasets'). &lt;abend reason&gt;..."), and an **error**-level log line so it appears in the Zowe MCP output window; it also sets `abendDetected: true` on the Native backend error log entry. The **MCP error** returned to the user explains that an unexpected internal Zowe Native (z/OS) error occurred, with the abend message (e.g. CEE3204S) as detail. **Stderr abend capture (immediate fail)**: Because the SDK may log "Invalid JSON response: CEE3204S..." to stderr without rejecting the in-flight Promise (ZNP-010), the server installs a temporary `process.stderr.write` wrapper during each ZNP request. When output matching that abend pattern is seen, the wrapper calls a callback that rejects a Promise that is raced with the request and the timeout, so the request **fails immediately** (no timeout wait). The snippet is also stored for the timeout path. See `stderr-abend-capture.ts`. The client always gets the user-facing error (either from that reject or, on timeout, by replacing the timeout error with the captured snippet and then the same user-facing message). The current MCP tool name is provided via **AsyncLocalStorage** (`mcp-tool-context.ts`); tool-call logging runs each handler inside `runWithMcpTool(name, ...)` so the backend can read it when an abend occurs. **CEEDUMP collection**: When an abend is detected the server fires a background task that opens a new SSH session, resolves the user's USS home via `uss.issueCmd('echo $HOME')`, lists files for `CEEDUMP*`, reads the newest by mtime, and writes a single file `zowe-ceedump-<timestamp>-<host>-<user>.txt` with metadata in YAML at the top (js-yaml), then `---` on a line, then the dump content. Save directory: `ZOWE_MCP_CEEDUMP_SAVE_DIR` if set, else `ZOWE_MCP_WORKSPACE_DIR` (set by the VS Code extension to the first workspace folder), else `process.cwd()`. The connection used for collection is evicted afterward. When the extension pipe is connected, after saving the file the server sends a **ceedump-collected** event with `path`, `reason` (abend text), `znpOperation`, and `mcpTool`; the extension shows an information message with that context and an "Open Dump" button that opens the dump in the editor, and logs a line to the Zowe MCP output channel with the full path.

## Eval-Driven Improvement Methodology

The project uses a data-driven approach to validate tool definition changes. Every proposed change to tool names, descriptions, or parameters is treated as a hypothesis tested with before/after eval runs.

### Workflow

1. **Baseline**: Run `npm run eval-compare -- --set naming-stress,description-quality --label "baseline"` to establish pass rates.
2. **Change**: Make one targeted change (e.g. improve a parameter description).
3. **Re-measure**: Run eval-compare with a new label (e.g. `--label "after-<change-name>"`).
4. **Compare**: Check `docs/eval-scoreboard.md` for the delta. Keep changes that improve or maintain pass rates; revert regressions.

### eval-compare Tool

`npm run eval-compare` (`packages/zowe-mcp-evals/src/eval-compare.ts`) runs evals across one or more models and auto-updates `docs/eval-scoreboard.md`. Key options: `--set` (comma-separated set names or `all`), `--model` (comma-separated model IDs from `evals.config.json` or `all`), `--label` (human-readable tag), `--repetitions` (override set default), `--no-cache` (disable LLM result cache). Reports go to `evals-report/<label>-<timestamp>/`. **LLM result cache**: Both `npm run evals` and `npm run eval-compare` cache LLM run results (tool calls + final text) in `.evals-cache/` keyed by a stable hash of system prompt, question prompt, tool definitions, and model ID. When a cache hit occurs, the cached result is replayed against the current assertions (no LLM call). Cache entries are written only when the question's pass rate meets `minSuccessRate`. Use `--no-cache` to force fresh LLM calls for every run. Cache stats (hits, writes, LLM calls) are reported in the console output and in `comparison.md`.

### Stress-Test Question Sets

- **naming-stress.yaml** (14 questions): CLI phrasing ("List data sets matching..."), z/OS jargon ("Allocate a PDS with RECFM=FB"), ISPF vocabulary ("Show me all data sets under USER like ISPF 3.4"), ambiguous natural language ("Find all occurrences of PERFORM"), cross-domain terminology ("spool files", "grep", "ls"). Config: `repetitions: 10`, `minSuccessRate: 0.5`, mock default preset.
- **sms-allocation.yaml** (4 questions): SMS and allocation parameter mapping — VOLSER, DATACLAS, STORCLAS, MGMTCLAS, JCL-style allocation. Validates that the agent maps z/OS SMS vocabulary to `createDataset` parameters. Config: `repetitions: 10`, `minSuccessRate: 0.5`, mock default preset.
- **description-quality.yaml** (11 questions): Pagination awareness (exact member count requiring multiple pages), search option combinations (caseSensitive, cobol, ignoreSequenceNumbers, includeContextLines), read windowing (first N lines, specific range), dataset attributes. Config: `repetitions: 10`, `minSuccessRate: 0.5`, mock pagination preset.

### Key Findings

- **Parameter descriptions matter more than parameter names for LLM tool use.** Renaming `string` to `searchString` in `searchInDataset` caused a slight regression (100% → 94.4% on naming-stress) because the model reads `.describe()` text, not just the key name. The original description "Search string (literal) to find in the data set or members" was already sufficient context.
- **Expanding z/OS jargon in descriptions helps.** Adding full names like "Record Format (RECFM)", "Logical Record Length (LRECL) in bytes", "Physical Sequential — a flat file" to `.describe()` strings improved description-quality from 63.6% to 72.7% (+9.1%).
- **Persistent failure areas**: pagination awareness (model doesn't always fetch all pages), `ignoreSequenceNumbers` vs `cobol` confusion (both relate to column handling), combined search options (model drops parameters when combining).
